# Data Contract

This document defines which files are yours and which are part of the system. It protects your personal data during skill updates.

---

## Why This Matters

When you update the `dossier.skill` package, the system layer (skill definition, governance docs, README) may be replaced with newer versions. Your vault layer (CV, profile, evals, outreach, all working files) is never touched — it stays yours.

This contract explicitly lists what is what, so you can update Dossier with confidence that your job-search data won't be overwritten.

---

## User Layer — Never Overwritten

These files are **100% yours**. The skill will never modify them, even on a major version update.

```
Dossier/
├── cv.md                      # Your work history. Sacred.
├── profile.md                 # Your target roles and signals. Sacred.
├── config.md                  # Your configuration (if present). Sacred.
├── stories.md                 # Your reusable story library (if present). Sacred.
├── dashboard.md               # Your Dataview queries for Obsidian. Sacred.
│
├── evals/                     # All your company evaluations. Sacred.
├── outreach/                  # All your outreach drafts and sends. Sacred.
├── cover-letters/             # All your cover letters. Sacred.
├── interview-prep/            # All your interview prep docs. Sacred.
├── research/                  # All your company research and deep dives. Sacred.
├── daily/                     # All your daily scans and leads. Sacred.
├── weekly/                    # All your weekly digests. Sacred.
└── archive/                   # All your archived companies. Sacred.
```

**Why:**
- These files contain your strategy, your story, your history.
- If they were overwritten, you'd lose months of work.
- The skill's job is to help you create and manage these, not to own them.

**Your responsibility:**
- Keep these files under version control (Git, Synology, whatever you prefer).
- Back them up regularly.
- Never place API keys or passwords in these files.

---

## System Layer — Updatable (Safe to Overwrite)

These files are part of the Dossier system. When you update the skill, these may be replaced with newer versions.

```
Dossier/
├── dossier.skill              # The skill package (ZIP archive). Updatable.
├── PRIVACY.md                 # Privacy and threat model. Updatable.
├── DATA_CONTRACT.md           # This file. Updatable.
├── README.md                  # Vault layout and conventions. Updatable.
└── LICENSE                    # Open-source license. Updatable.
```

**What's inside `dossier.skill`:**

`dossier.skill` is a ZIP archive containing:
- `SKILL.md` — the skill definition (prompt, inputs, outputs, metadata)
- `scoring-guide.md` — detailed scoring rubric for company evaluations

These files live *inside* the package, not as standalone files in the vault. You don't normally need to inspect them, but if you do:

```bash
unzip dossier.skill -d /tmp/skill-inspect/
cat /tmp/skill-inspect/SKILL.md
```

**When updates happen:**
- A new `dossier.skill` may be published with improvements to scoring, new evaluation sections, or better prompts.
- A new `README.md` may document new folder conventions or archival rules.
- A new `PRIVACY.md` may reflect new integrations or threat scenarios.
- Your `.md` files (cv.md, profile.md, etc.) are never touched.

**What to expect:**
- `dossier.skill` updates are safe; they won't overwrite your data.
- If you've customized `PRIVACY.md`, `DATA_CONTRACT.md`, or `README.md`, you may want to merge your changes with the new version (we'll provide guidance on conflict resolution when this happens).

**Your responsibility:**
- Review system file updates to understand what changed.
- If you've added custom sections to README.md or PRIVACY.md, be prepared to re-add them after an update.
- Keep the skill version updated so you get the latest fixes and features.

## Derived Files — Created by Skill, Owned by You

These files are generated by the skill at your request. Once created, they're yours to keep, modify, and manage.

```
Dossier/
├── cv-[slug]-[date].md        # Tailored CV (e.g., cv-acme-corp-2026-01-15.md). Updatable only on your request.
├── cv-[slug]-[date].docx      # ATS-formatted export (e.g., cv-acme-corp-2026-01-15.docx). Updatable only on your request.
└── [other custom outputs]     # Negotiation briefs, interview guides, etc. Updatable only on your request.
```

**Lifecycle:**
1. You ask the skill: "Tailor my CV for Acme Corp."
2. Skill creates `cv-acme-corp-2026-01-15.md` and `cv-acme-corp-2026-01-15.docx`.
3. These files are now yours. You edit them, use them, send them.
4. The skill will not overwrite them unless you explicitly ask.
5. If you ask to tailor again, the skill may offer to update in place or create a new dated version.

**Your responsibility:**
- Treat these files like any other work product — review them before using.
- Manually edit them if needed (e.g., fix a date, add a metric, change a sentence).
- Version them if you're tracking iterations (the `[date]` in the filename helps).

## Notion (Optional Integration)

If you configure Notion as your pipeline tracker, Notion data is separate from the vault:

- **Vault files** (evals, outreach, etc.) are yours and never touched by updates.
- **Notion database** (pipeline state, grades, statuses) is synchronized when you use the skill.
- If you're not using Notion, this doesn't apply.

For more details, see [[PRIVACY|PRIVACY.md]].

## Summary Table

| Category | Files | Updated? | Your Responsibility |
|----------|-------|----------|---------------------|
| **User Layer** | cv.md, profile.md, all evals, outreach, prep, research, daily, weekly, archive | Never | Back up, keep in version control, don't store credentials |
| **System Layer** | dossier.skill, PRIVACY.md, DATA_CONTRACT.md, README.md, LICENSE | Automatically on skill update | Review changes, merge custom edits if any |
| **Derived Files** | cv-[slug]-[date].md, cv-[slug]-[date].docx, etc. | Only on explicit request | Review before use, edit as needed, version as you see fit |

## Questions?

- **"Will my CV.md be overwritten?"** No, never. cv.md is user layer.
- **"What if I customize README.md?"** It's in the system layer, so it may be overwritten on updates. Keep your customizations in a separate `README-custom.md` or make a note of them.
- **"Can I trust the skill with my personal data?"** Your vault is local-first. The skill only sees what you explicitly share with it in a conversation. [[PRIVACY|PRIVACY.md]] has the full threat model.
- **"What happens if I update the skill?"** Your `.md` files stay. System files may be replaced. Derived files are safe unless you ask for a re-tailor.
